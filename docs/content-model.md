# Kubtel.ru: контентная модель этапа 1

Документ описывает рабочий контракт локального content layer. Его цель - не дать черновым коммерческим данным выглядеть как подтвержденное предложение и подготовить структуру для будущей CMS.

## Базовые статусы

| Статус               | Когда использовать                                                                                                  | Можно ли подавать как факт        |
| -------------------- | ------------------------------------------------------------------------------------------------------------------- | --------------------------------- |
| `confirmed`          | Данные подтверждены ответственным лицом Kubtel, договором, выгрузкой CRM/биллинга или надежным публичным источником | Да                                |
| `needs_verification` | Смысл или сценарий выглядит правдоподобно, но требует сверки                                                        | Только с осторожной формулировкой |
| `draft`              | Редакционная или продуктовая гипотеза для прототипа                                                                 | Нет                               |

## Общие поля источника

Каждая контентная сущность получает `contentSource`.

| Поле          | Назначение                                                                                                |
| ------------- | --------------------------------------------------------------------------------------------------------- |
| `status`      | Общий статус достоверности сущности                                                                       |
| `type`        | Тип источника: команда Kubtel, публичный сайт, legacy-данные, технический аудит или редакционная гипотеза |
| `label`       | Человеческое название источника                                                                           |
| `checkedAt`   | Дата последней сверки или `null`, если сверки не было                                                     |
| `responsible` | Роль владельца подтверждения: `commercial`, `operations`, `coverage`, `content`                           |
| `note`        | Что именно нужно проверить или почему статус не подтвержден                                               |

## Коллекции

| Коллекция  | Назначение                               | Ключевые поля проверки                       |
| ---------- | ---------------------------------------- | -------------------------------------------- |
| `tariffs`  | Тарифные карточки и расчет стоимости     | `contentSource`, `commercialReview`, `proof` |
| `services` | Описание услуг и дополнительных опций    | `facts`, `contentSource`                     |
| `faq`      | Ответы на пользовательские тревоги       | `proof`, `contentSource`                     |
| `coverage` | Город, районы, улицы, ЖК, частный сектор | `connectionStatus`, `contentSource`          |
| `promos`   | Акции и промо-механики                   | `proof`, `contentSource`                     |

## Коммерческий блок тарифа

Для тарифов используется отдельный `commercialReview`, потому что цена, скорость, опции и подключение могут подтверждаться разными источниками.

| Поле               | Что проверяет                                          |
| ------------------ | ------------------------------------------------------ |
| `status`           | Общий статус коммерческой готовности тарифа            |
| `priceStatus`      | Абонентская плата, промо-цена, цена после промо        |
| `speedStatus`      | Скорость загрузки и отдачи                             |
| `optionsStatus`    | Роутер, статический IP, ТВ-пакеты и другие опции       |
| `connectionStatus` | Стоимость и условия подключения                        |
| `requiredEvidence` | Список документов или выгрузок, которые нужно получить |
| `note`             | Ограничения публикации                                 |

## Правила публикации

1. Тариф со статусом `draft` нельзя показывать как финальное коммерческое предложение.
2. Акцию со статусом `draft` нельзя выводить без явной пометки и условий.
3. FAQ со статусом `needs_verification` должен отвечать осторожно: через процесс, а не через неподтвержденное обещание.
4. Зону покрытия со статусом `draft` нельзя превращать в автоматическое "доступно"; адрес должен уходить на ручную проверку.
5. Любое сильное утверждение в интерфейсе должно иметь `proof` или `contentSource` со статусом не ниже `needs_verification`.

## Подготовка к CMS

Текущие JSON-файлы должны переноситься в CMS как отдельные модели без изменения контрактов UI:

- `Tariff`;
- `Service`;
- `FaqItem`;
- `CoverageArea`;
- `Promo`;
- общий вложенный тип `ContentSource`;
- вложенный тип `CommercialReview` для тарифов.

Компоненты должны читать данные через эти поля и не зависеть от того, пришли они из локального JSON, Headless CMS или внутреннего API.
