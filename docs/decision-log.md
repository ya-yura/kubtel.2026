# Kubtel.ru: журнал решений

Этот документ фиксирует архитектурные, продуктовые и процессные решения. Решение можно изменить, но новое решение должно ссылаться на предыдущее и объяснять причину изменения.

## ADR-001: вести проект через фиксируемые этапы и чекапы

- Дата: 2026-05-06.
- Статус: accepted.
- Контекст: проект должен развиваться последовательно, без постоянного возврата к уже закрытым вопросам, но с возможностью менять ранее реализованное при появлении новых данных.
- Решение: вести `docs/project-state.md` как текущую память проекта, `docs/implementation-log.md` как хронологию работ и `docs/decision-log.md` как журнал решений.
- Последствия: каждый завершенный этап получает явную отметку, список результатов, проверки и следующий шаг.

## ADR-002: стартовать с локального контентного слоя

- Дата: 2026-05-06.
- Статус: accepted.
- Контекст: пока нет подтвержденного доступа к Headless CMS, CRM, биллингу и актуальной базе тарифов.
- Решение: первую техническую версию строить на локальных структурированных данных и TypeScript-интерфейсах. Компоненты не должны зависеть от конкретного источника данных.
- Последствия: позже можно подключить Strapi, Sanity, Storyblok или внутренний API без полной переработки интерфейса.

## ADR-003: фиксировать локальный dev-сервер на 127.0.0.1:4321

- Дата: 2026-05-06.
- Статус: accepted.
- Контекст: первый запуск Astro слушал IPv6 `::1`, что мешало проверке через `127.0.0.1`.
- Решение: в `package.json` скрипт `dev` явно запускает Astro с `--host 127.0.0.1 --port 4321`.
- Последствия: следующие итерации интерфейса можно проверять по стабильному адресу `http://127.0.0.1:4321/`.

## ADR-004: отделять источник достоверности от текста и коммерческих полей

- Дата: 2026-05-06.
- Статус: accepted.
- Контекст: на старте нет подтвержденной тарифной матрицы, адресной базы, SLA и юридических формулировок, но интерфейс уже должен строиться на реалистичных сценариях.
- Решение: ввести общий `contentSource` для всех контентных сущностей и отдельный `commercialReview` для тарифов. Черновые данные маркировать как `draft`, данные на сверке - как `needs_verification`.
- Последствия: UI может развиваться без риска выдать гипотезу за факт; при появлении подтвержденной выгрузки достаточно обновить content layer, не меняя контракты компонентов.

## ADR-005: вести сайт Kubtel как отдельный git-репозиторий

- Дата: 2026-05-06.
- Статус: accepted.
- Контекст: рабочая папка сайта находится внутри большого родительского git-корня `O:\Dev`, где есть много посторонних проектов и файлов.
- Решение: инициализировать отдельный git-репозиторий в `O:\Dev\kubtel-best-redesign`, подключить `origin` к `https://github.com/ya-yura/kubtel.2026.git` и пушить логические контрольные точки.
- Последствия: история сайта изолирована от родительской директории, а дальнейшая работа может фиксироваться регулярными коммитами и push.
